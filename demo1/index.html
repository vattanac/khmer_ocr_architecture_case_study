Diagram 2 — Why Khmer is Hard: The 5-Row Vertical Structure

A single Khmer syllable can occupy up to 5 vertical rows simultaneously. This is what makes old OCR methods fail.

Row 1 — Superscript
◌់
Tiny diacritics (~15% height)
Row 2 — Above-base Vowel
◌ី ◌ឹ ◌ើ
Above vowels (~25% height)
Row 3 — Base Consonant
ក គ ស ន
Main character (100% size)
Row 4 — Subscript Consonant (Coeng)
្ត ្រ ្ន
Below consonants (~60% height)
Row 5 — Below-base Vowel
◌ុ ◌ូ
Below vowels (~25% height)

Example: "ស្ត្រី" (woman)

ស្ត្រី
ស — Base consonant (Row 3)
្ត — Subscript "ta" below ស (Row 4)
្រ — Subscript "ra" below ្ត (Row 4, deeper)
ី — Above vowel "ii" above ស (Row 2)
Key insight: This single word uses 4 of 5 vertical rows. A computer must identify ALL components and their relationships correctly.

Diagram 4 — Detailed Model Architecture (Stage by Stage)

The complete pipeline from raw image to Unicode text output, with dimensions and operations at each step.

IN
Input

Input Image

A cropped text-line image containing Khmer text. Resized to a fixed height of 64 pixels while preserving the aspect ratio.

Image shape: H=64 × W=variable × C=3 (RGB)
Example: 64 × 512 × 3 for a medium-length line
Example: 64 × 2000 × 3 for a very long line → triggers chunking
Why 64px? Testing showed 32px loses tiny diacritics (12.1% CER), while 64px captures all 5 vertical rows clearly (3.1% CER). Going above 64px gives no improvement but costs more memory.
CNN
Feature Extraction

Stage 1 — CNN Backbone (ResNet-45)

Scans the image with convolutional filters to extract visual features at multiple levels of abstraction.

Process:
Input: 64 × 512 × 3
→ Conv Block 1: detect edges, corners → 32 × 256 × 64
→ Conv Block 2: detect strokes, curves → 16 × 128 × 128
→ Conv Block 3: detect character parts → 8 × 64 × 256
→ Conv Block 4: detect full character shapes → 8 × 64 × 512

Output Feature Map: H=8 × W=64 × C=512

The height of 8 rows still preserves the spatial positions of all 5 Khmer vertical zones. This is why using 2D features is critical.

P
Patching

Stage 2 — Adaptive 2D Patching Module

Converts the 2D feature map into a sequence of patches that the Transformer can process, while preserving all vertical information.

Process:
Input Feature Map: 8 × 64 × 512

→ Divide into column-wise patches
→ Each patch = one column across ALL 8 rows
→ Each patch shape: [8 × 1 × 512]
→ Flatten each patch: [8 × 512] = 4096-dim vector
→ Linear projection to model dim: 4096 → 512-dim
→ Add positional encoding (tells model WHERE each patch is)

Output: 64 patch vectors, each 512-dim
Why this matters: Unlike 1D methods that collapse rows 1-5 into one number, each patch-vector here has separate positions for Row 1 (diacritics), Row 3 (base consonant), Row 4 (subscripts), etc. The Transformer can learn to "look at" specific rows within each patch.
E
Encoder

Stage 3 — Transformer Encoder (12 layers)

Processes all 64 patches simultaneously with self-attention, allowing each patch to "communicate" with every other patch.

Architecture:
Layers: 12
Attention heads per layer: 8
Model dimension: 512
Feed-forward dimension: 2048
Dropout: 0.1

What each layer does:
1. Multi-Head Self-Attention:
Each patch asks "which other patches are relevant to me?"
Example: vowel patch above ស attends to the ស consonant patch

2. Feed-Forward Network:
Refines each patch's representation independently

3. Layer Normalization + Residual Connection:
Stabilizes training, prevents information loss

Output: 64 enriched patch vectors, each 512-dim
(Same shape as input, but now each patch "knows about" all other patches)
D
Decoder

Stage 4 — Transformer Decoder (Auto-regressive)

Generates the output text one Unicode character at a time, attending to both previously generated characters and encoder output.

Step-by-step generation:

Step 0: Input = [START]
→ Attend to encoder output (all 64 patches)
→ Predict: "ស" (highest probability)

Step 1: Input = [START, ស]
→ Attend to encoder + previous character
→ Predict: "្" (coeng sign)

Step 2: Input = [START, ស, ្]
→ Predict: "ត"

Step 3: Input = [START, ស, ្, ត, ្]
→ Predict: "រ"

Step 4: Input = [START, ស, ្, ត, ្, រ]
→ Predict: "ី"

Step 5: Input = [START, ស, ្, ត, ្, រ, ី]
→ Predict: [END]

Final output: "ស្ត្រី" ✓
Vocabulary size: ~120 tokens (114 Khmer Unicode codepoints + special tokens [PAD], [UNK], [START], [END])
OUT
Output

Final Output — Khmer Unicode Text

The decoded Unicode string can now be stored, searched, translated, or spoken aloud by a computer.

Diagram 11 — Known Failure Cases

Where the system still struggles and why.

1. Low Resolution

When images are below 40px tall, tiny diacritics become invisible. The difference between ក and ក់ (a single dot) disappears.

2. Text Detector Errors

If the upstream text detector crops too tightly, Row 1 diacritics above the text get cut off. The recognizer never sees them.

3. Ambiguous Characters

Some pairs like ◌ិ vs ◌ី differ by just one stroke length. At low quality, even humans struggle to distinguish them.

4. Rare Triple Clusters

Triple-stacked consonants (base + 2 subscripts) are rare in training data. The model sometimes drops the deepest subscript.

5. Historical Script Variation

Ancient palm-leaf manuscripts use character forms very different from modern fonts. Without fine-tuning, error rates jump significantly.
